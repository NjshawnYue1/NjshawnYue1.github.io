Spring-Security

简单使用

在一个最简单的web的springboot的应用中，用户可以根据url直接获取到结果，如直接访问http://localhost:8080/hello

但当我们引入spring-security依赖时，会对这个请求做一个保护，在什么都不需要配置的情况下，用户将不能直接对这个url进行访问而需要认证以及授权。这个依赖的自动配置文件默认为我们提供了一个登陆用户名为user密码为随机生成的一个账号。在springboot 2.x版本中，会直接跳转至一个登陆页面，这个页面由依赖提供。

集成相关依赖后，进行简单的配置不需要编写任何配置文件就可以使用。

spring-security核心组件

SecurityContext

用户通过安全验证后，相关的验证信息会存储在这个对象中，通过此方法可以拿到认证对象。

SecurityContextHolder

在典型的web应用程序中，用户登录一次，然后由其会话ID标识。服务器缓存持续时间会话的主体信息。在Spring Security中，在请求之间存储SecurityContext的责任落在SecurityContextPersistenceFilter上，默认情况下，该上下文将上下文存储为HTTP请求之间的HttpSession属性。它会为每个请求恢复上下文SecurityContextHolder，并且最重要的是，在请求完成时清除SecurityContextHolder。SecurityContextHolder是一个类，他的功能方法都是静态的（static）。

Authentication

authentication 直译过来是“认证”的意思，在Spring Security 中Authentication用来表示当前用户是谁，一般来讲你可以理解为authentication就是一组用户名密码信息。Authentication也是一个接口，其定义如下：

接口有4个get方法，分别获取

  • Authorities, 填充的是用户角色信息。
  • Credentials，直译，证书。填充的是密码。
  • Details ，用户信息。
  • ,Principal 直译，形容词是“主要的，最重要的”，名词是“负责人，资本，本金”。感觉很别扭，所以，还是不翻译了，直接用原词principal来表示这个概念，其填充的是用户名。

因此可以推断其实现类有这4个属性。这几个方法作用如下：

  • getAuthorities: 获取用户权限，一般情况下获取到的是用户的角色信息。
  • getCredentials: 获取证明用户认证的信息，通常情况下获取到的是密码等信息。
  • getDetails: 获取用户的额外信息，（这部分信息可以是我们的用户表中的信息）
  • getPrincipal: 获取用户身份信息，在未认证的情况下获取到的是用户名，在已认证的情况下获取到的是 UserDetails (UserDetails也是一个接口，里边的方法有getUsername,getPassword等)。
  • isAuthenticated: 获取当前 Authentication 是否已认证。
  • setAuthenticated: 设置当前 Authentication 是否已认证（true or false）。

方法含义如下：

UserDetailsService

提到了UserDetails就必须得提到UserDetailsService, UserDetailsService也是一个接口，且只有一个方法loadUserByUsername，他可以用来获取UserDetails。

通常在spring security应用中，我们会自定义一个CustomUserDetailsService来实现UserDetailsService接口，并实现其public UserDetails loadUserByUsername(final String login);方法。我们在实现loadUserByUsername方法的时候，就可以通过查询数据库（或者是缓存、或者是其他的存储形式）来获取用户信息，然后组装成一个UserDetails,(通常是一个org.springframework.security.core.userdetails.User，它继承自UserDetails) 并返回。

在实现loadUserByUsername方法的时候，如果我们通过查库没有查到相关记录，需要抛出一个异常来告诉spring security来“善后”。这个异常是org.springframework.security.core.userdetails.UsernameNotFoundException。

作用域

全局作用域只有一个，每个函数又都有作用域（环境）。这也是为什么在

show函数中访问的this是Lesson对象，而在map函数中this又变成了window。

作用域的访问关系

在内部作用域中可以访问到外部作用域的变量，在外部作用域中无法访问到内部作用域的变量。

变量的提升(声明提前)

使用var关键字声明的变量（ 比如 var a = 1），会在所有的代码执行之前被声明（但是不会赋值），但是如果声明变量时不是用var关键字（比如直接写a = 1），则变量不会被声明提前。如果先使用还会报引用错误

为什么会有变量提升呢，在js运行时会有一个预编译的过程。

规律1：任何变量，如果未经声明就赋值，此变量是属于 window 的属性，而且不会做变量提升。（注意，无论在哪个作用域内赋值）

规律2：一切声明的全局变量，全是window的属性。（注意，我说的是在全局作用域内声明的全局变量，不是说局部变量）

未被声明的变量属于window但不会被提升，声明的全局变量属于window会被提升。

所有已生成的时间，无论是用户的点击还是键盘按压或者是服务器端的ajax事件都会被放在一个事件队列中。以他们被浏览器检测到的顺序进行排列。浏览器回循环检查时间的队列头如果有事件则会进行处理。

事件处理器是指当某个事件发生之后，我们希望执行的函数。想要达到这个目标，最关键的是知道什么样的事件发生了。也就是需要为这个事件注册一个处理器。

1.通过把函数赋给某一个属性 windown.load = function(){}

2.通过使用内置的addEventListener方法

spring IOC

底层原理

为什么需要依赖注入，程序之间的紧耦合导致程序无法被扩展，或者在扩展时很不方便，需要修改程序的源代码，不符合开放关闭原则，比如，如果需要在控制层实现业务逻辑，传统的做法是在控制层直接new出service层的对象再调用其中的方法，但是这就导致两个类之间的耦合程度变高，如何解耦呢？因为是需要service层不同的对象，所以可以使用工厂模式来提供不同的实现类，但在这种情况下，控制层又和工厂类紧耦合了，为了进一步解耦，提供的方法是：工厂方法+反射+配置文件。

此时这个工厂类就是spring工厂，spring容器，我们需要什么类直接先获取这个容器在利用反射机制得到这个类(getBean()),在获取时需要强制类型转换，在操作这个类的对象即可。这就是IOC的底层原理。也就是本来需要手动创建的对象现在交给spring进行管理。需要的对象直接从容器中通过id获取即可。

DI依赖注入

将传统方式的set方法设置属性取消，本身也是需要创建对应的实现类对象才可以调用set方法，如果new的是接口对象则没有set方法这样如果类中带有属性，传统的方式就失去了意义，不面向接口就没有扩展性可言。而使用依赖注入则只需要在配置文件中设置它的属性即可，基本数据类型通常使用property标签即可。这样在容器中获取对象是这个属性就已经在这个对象中被设置好了。

spring的工厂类

实际上就是通过ClassPathXmlApplicationContext（ApplicationContext），与BeanFactory的区别在于，部分功能有扩展以及实例化bean的实际不同，一个是在调用getBean()方法的时候才会获取，一个则是在配置文件被加载的时候就会实例化所有单例的bean。
